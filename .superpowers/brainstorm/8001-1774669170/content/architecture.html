Architecture — New & Changed Files

What gets added or modified. Everything else stays untouched.

🆕 New files
src/app/api/extract-resume/route.ts

POST handler. Accepts multipart form upload, reads file buffer, runs pdf-parse / mammoth / plain-text depending on mime type, returns { text }. Auth-gated.

src/components/onboarding/ResumeImportStep.tsx

Client component. Drag-and-drop + file picker UI. Manages the upload → extract → parse pipeline. Shows loading states. On success, hands ParsedResume up to parent.

src/components/onboarding/ParsedResumeReview.tsx

Client component. Displays parsed data summary (name, headline, counts). "Import to Profile" calls saveProfile and redirects. "Edit first" redirects to /dashboard/profile with data in sessionStorage. "Start over" resets.

✏️ Modified files
src/app/onboarding/OnboardingClient.tsx

Add a new step before the existing profile completion step. Shows the two-path choice (import vs. scratch). Renders ResumeImportStep or calls the existing completion logic.

src/lib/actions/profile.ts

No logic changes. The existing saveProfile() action is called directly from ParsedResumeReview — it already handles everything we need.

package.json

Two new dependencies: pdf-parse (PDF text extraction) and mammoth (DOCX → text). Both are small, pure Node.js libraries.

Data flow (import path)
OnboardingClient
  → ResumeImportStep (file selected)
    → POST /api/extract-resume → { text }
    → POST /api/parse-resume → ParsedResume
  → ParsedResumeReview (data shown)
    → saveProfile(data) [server action]
    → redirect /dashboard/profile
"Edit first" path: ParsedResumeReview stores parsed data in sessionStorage under a known key, then redirects to /dashboard/profile. The profile page checks for that key on load and pre-populates the form — without saving. User reviews and hits "Save changes" themselves.
